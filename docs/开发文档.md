# GloscAI Images APP 开发文档

日期：2026-06-15  
资料来源：README.md

## 1. 当前仓库状态

当前仓库仅包含 README.md，尚未包含 Android 工程文件、Gradle 配置或源码目录。

本开发文档以 README 中明确的技术栈为基础，给出推荐开发规范、工程结构、实现步骤和验证流程。创建 Android 工程后，应根据实际源码更新本文档中的包名、模块名和命令。

## 2. 技术栈

README 已明确技术栈：

- Android 原生开发：Java/Kotlin。
- OpenAI API：用于生成图片。
- Room：用于图片管理和本地数据存储。
- Retrofit：用于网络请求。
- Glide：用于图片加载。

建议补充：

- Kotlin Coroutines：处理异步任务。
- OkHttp：配合 Retrofit 处理拦截器、超时和日志。
- WorkManager：处理长耗时生成任务或后台恢复。
- AndroidX Lifecycle：ViewModel 和状态管理。
- Android Keystore 或 EncryptedSharedPreferences：保护 API Key。

## 3. 开发环境建议

- Android Studio：使用稳定版。
- JDK：建议 JDK 17。
- Gradle：以 Android Studio 创建工程时生成的版本为准。
- Kotlin：如采用 Kotlin，使用与 Android Gradle Plugin 兼容的版本。
- 设备：Android 真机或模拟器。

待确认：

- minSdk。
- targetSdk。
- 应用包名。
- UI 技术选型：XML View、Jetpack Compose 或混合。

## 4. 工程初始化建议

如尚未创建 Android 工程，建议先创建原生 Android App：

1. 使用 Android Studio 创建 Empty Activity 项目。
2. 语言优先选择 Kotlin，如团队已有 Java 代码规范，可使用 Java 或混合。
3. 包名建议使用公司或产品域名反写，例如 `com.glosc.images`。
4. 初始化 Git 后确认 `.gitignore` 已排除构建产物和本地密钥文件。
5. 创建 `docs/` 目录保存项目、设计和开发文档。

## 5. 推荐目录结构

单 module 初期结构：

```text
app/src/main/java/com/glosc/images/
  ui/
    generate/
    chat/
    edit/
    library/
    settings/
  domain/
    model/
    usecase/
  data/
    api/
    db/
    repository/
    storage/
  core/
    common/
    image/
    network/
    security/
```

资源目录建议：

```text
app/src/main/res/
  drawable/
  mipmap/
  values/
  layout/
  navigation/
```

如采用 Jetpack Compose，`layout/` 可减少使用，但仍应保留主题、图标和字符串资源管理。

## 6. 依赖配置建议

需要加入的核心依赖：

- Retrofit。
- OkHttp。
- Room Runtime。
- Room Compiler 或 KSP。
- Glide。
- AndroidX Lifecycle ViewModel。
- Kotlin Coroutines。

注意事项：

- 使用 Room 时需要配置 schema 导出目录。
- 使用 Retrofit 时需要明确 JSON 解析库，例如 Moshi 或 Gson。
- 使用 Glide 时注意大图内存占用和缩略图加载策略。
- 不要将 API Key 写入 Gradle、源码或 README。

## 7. 配置管理

### 7.1 API Key

开发阶段可以通过应用内设置页输入 API Key。

建议：

- 明文 API Key 不落入 Git。
- 本地存储使用 Android Keystore 或 EncryptedSharedPreferences。
- 日志中屏蔽 Authorization header。
- API Key 缺失时阻止生成请求并给出设置入口。

### 7.2 自定义 API

建议配置项：

- 服务名称。
- Base URL。
- API Key。
- 默认模型。
- 请求超时时间。
- 是否启用。

配置保存后应提供连通性测试，避免用户进入生图流程后才发现配置错误。

### 8.9 设置页

## 9. 开发规范

### 9.1 命名

- 类名使用 PascalCase。
- 方法和变量使用 camelCase。
- Room Entity 后缀建议使用 `Entity`。
- DAO 后缀使用 `Dao`。
- Repository 后缀使用 `Repository`。
- Use Case 后缀使用 `UseCase`。

### 9.2 状态管理

建议统一页面状态：

```kotlin
sealed interface UiState<out T> {
    data object Idle : UiState<Nothing>
    data object Loading : UiState<Nothing>
    data class Success<T>(val data: T) : UiState<T>
    data class Error(val message: String, val cause: Throwable? = null) : UiState<Nothing>
}
```

### 9.3 错误处理

- 网络层错误不直接暴露给 UI。
- Repository 将底层异常转换为业务错误。
- UI 展示用户可理解的错误信息。
- 失败任务保留在数据库中，便于重试和排查。

### 9.4 日志

- Debug 构建可输出请求路径和任务 ID。
- 禁止输出 API Key、Authorization header 和完整敏感请求体。
- Release 构建关闭详细网络日志。

## 10. 构建与运行

Android 工程创建后，可使用以下命令：

```bash
./gradlew assembleDebug
./gradlew installDebug
./gradlew test
./gradlew connectedAndroidTest
```

当前仓库尚未包含 Gradle 工程，因此以上命令需要在工程初始化后执行。

## 11. 测试策略

- 请求参数构造。
- API 错误映射。
- Repository 行为。
- Use Case 状态流。

- 任务状态更新。

### 11.3 UI 测试

优先覆盖：

- 生图表单提交。
- 生成结果展示。
- 图片库搜索和筛选。
- 图片删除确认。

### 11.4 手工测试

每次发布前检查：

- 新安装启动。
- API Key 配置。
- 成功生成图片。
- 网络断开后的错误提示。
