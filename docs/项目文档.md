# GloscAI Images APP 项目文档

日期：2026-06-15  
资料来源：README.md

## 1. 项目概述

GloscAI Images APP 是一款 Android 原生 AI 生图应用，面向需要通过提示词、参数配置、对话交互和图片编辑完成图像创作的用户。

应用核心能力包括：

- 工程式生图：用户配置参数和提示词后发起图片生成。
- 对话式生图：用户通过自然语言与 AI 对话，由 AI 理解上下文并生成图片。
- 图片编辑：支持图片拓展、修复、变换、局部修改等编辑操作。
- 图片变换：支持风格转换、图像增强、图像修复等处理能力。
- 自定义 API 接入：支持接入第三方 AI 服务或自建 AI 服务。
- 图片管理：支持对生成图片进行分类、标签和搜索。

## 2. 项目目标

### 2.1 产品目标

- 提供稳定、易用的移动端 AI 生图体验。
- 同时覆盖参数化创作和对话式创作两种主流程。
- 让用户能持续管理、复用和检索历史生成图片。
- 为不同 AI 服务商或自建服务预留扩展能力。

### 2.2 技术目标

- 使用 Android 原生技术栈实现，支持 Java/Kotlin 开发。
- 使用 Retrofit 封装网络请求，便于适配 OpenAI API 和自定义 API。
- 使用 Room 管理本地图片记录、标签、分类和任务状态。
- 使用 Glide 负责图片加载、缓存和展示。
- 保持模块边界清晰，便于后续扩展图片编辑、服务商适配和图片管理能力。

## 3. 目标用户

- 内容创作者：需要快速生成插画、封面、社媒图片或概念图。
- 设计和运营人员：需要通过模板化参数批量生成可用素材。
- 高级用户：需要配置自定义 API、模型参数和服务端地址。
- 个人开发者或团队用户：需要接入自建 AI 服务或第三方生图服务。

## 4. 功能范围

### 4.1 工程式生图

用户通过表单化界面配置图片生成参数。

建议覆盖：

- 正向提示词和负向提示词。
- 图片尺寸、数量、风格、质量、模型等参数。
- 随机种子或固定种子。
- 参数模板保存和复用。
- 生成中、成功、失败、取消等任务状态。

### 4.2 对话式生图

用户通过聊天界面描述创作需求，系统结合上下文生成图片。

建议覆盖：

- 多轮对话上下文。
- 文本消息和图片消息展示。
- 根据对话生成提示词或直接调用生图接口。
- 支持重新生成、继续修改和保存结果。

### 4.3 图片编辑

面向已有图片进行二次创作。

建议覆盖：

- 图片拓展。
- 图像修复。
- 图像变换。
- 局部修改。
- 编辑前后版本记录。

### 4.4 图片变换

对图片进行风格和质量处理。

建议覆盖：

- 风格转换。
- 图像增强。
- 图像修复。
- 变换参数配置。

### 4.5 自定义 API 接入

允许用户接入非默认服务。

建议覆盖：

- API Base URL 配置。
- API Key 配置。
- 模型名称配置。
- 请求头和参数映射配置。
- 服务连通性测试。

### 4.6 图片管理

为生成结果建立本地资产管理能力。

建议覆盖：

- 图片列表和详情。
- 分类、标签、搜索。
- 收藏、删除、导出和分享。
- 按生成时间、模型、标签和任务类型筛选。

## 5. 典型用户流程

### 5.1 工程式生图流程

1. 用户进入生图页面。
2. 输入提示词并选择参数。
3. 点击生成。
4. 应用调用 AI 生图 API。
5. 生成结果保存到本地数据库和文件存储。
6. 用户查看、编辑、收藏、分享或重新生成。

### 5.2 对话式生图流程

1. 用户进入对话页面。
2. 输入自然语言需求。
3. 应用记录对话上下文并请求 AI 服务。
4. AI 返回图片或生成任务结果。
5. 用户继续通过对话修改图片。
6. 满意后保存到图片库。

### 5.3 图片编辑流程

1. 用户从图片库选择图片。
2. 选择编辑方式。
3. 输入编辑要求或选择编辑区域。
4. 应用调用图片编辑接口。
5. 生成新版本并保存编辑历史。

## 6. 非功能需求

### 6.1 性能

- 图片列表应支持流畅滚动和懒加载。
- 大图加载应使用缩略图、缓存和占位状态。
- 长耗时生成任务应避免阻塞主线程。
- 网络请求应支持超时、重试和取消。

### 6.2 稳定性

- 网络失败、鉴权失败、余额不足、参数错误等状态需要明确提示。
- 生成任务状态需要可恢复，避免应用切后台后丢失结果。
- 本地数据库迁移需要版本化管理。

### 6.3 安全与隐私

- API Key 不应写入代码仓库。
- 本地保存的敏感配置建议使用 Android Keystore 或加密存储。
- 日志中不输出 API Key、完整鉴权头和敏感提示词。
- 用户删除图片时，应同时处理数据库记录和本地文件。

### 6.4 可扩展性

- AI 服务商接入层应与业务层解耦。
- 生成、编辑、变换应使用统一任务模型。
- 图片管理模块应能扩展标签、分类、相册和搜索能力。

## 8. 风险与待确认事项

- 具体 OpenAI 图片接口、模型和参数需要根据实际接入版本确认。
- 自定义 API 的协议兼容范围需要定义。
- 图片文件保存位置、缓存策略和清理策略需要明确。
- 最低 Android 版本、UI 技术选型和包名尚未在 README 中指定。
- 图片编辑功能依赖服务端能力，客户端需要为能力差异做降级设计。

## 9. 验收标准

- 用户可以配置 API Key 并成功发起一次图片生成。
- 用户可以查看生成结果，并在本地图片库中找到历史记录。
- 应用在网络失败、接口报错和参数缺失时有明确提示。
- 图片记录包含生成参数、时间、来源模型和本地文件地址。
- 自定义 API 配置不会影响默认 API 配置的可用性。
- 代码结构支持后续扩展对话式生图、图片编辑和图片管理功能。
